# Contributing

## Getting Started

1. Clone the repository
2. Run `pnpm install`
3. Copy `.env.example` to `.env` and configure
4. Run `pnpm dev` to start the development server

## Commit Messages

This project enforces [Conventional Commits](https://www.conventionalcommits.org/) via a commit-msg hook. Every commit message must follow this format:

```
type(scope): description
```

### Types

| Type       | When to use                                    |
| ---------- | ---------------------------------------------- |
| `feat`     | New feature or functionality                   |
| `fix`      | Bug fix                                        |
| `docs`     | Documentation only                             |
| `style`    | Formatting, semicolons, etc. (no logic change) |
| `refactor` | Code restructuring (no feature or fix)         |
| `perf`     | Performance improvement                        |
| `test`     | Adding or updating tests                       |
| `build`    | Build system or dependency changes             |
| `ci`       | CI/CD configuration                            |
| `chore`    | Maintenance tasks                              |
| `revert`   | Reverting a previous commit                    |

### Scope

The scope should identify the area of the codebase affected. Examples: `auth`, `user`, `health`, `prisma`, `deps`, `config`.

### Examples

```
feat(auth): add refresh token rotation
fix(health): return 503 when database is unreachable
docs(readme): add deployment section
refactor(user): move DTO mapping into service layer
chore(deps): update nestjs to 11.2
test(auth): add OIDC guard unit tests
build(vite): add source maps for staging builds
```

### Breaking Changes

For breaking changes, add `!` after the type/scope and include a `BREAKING CHANGE:` footer:

```
feat(auth)!: switch from session to stateless JWT

BREAKING CHANGE: All existing sessions are invalidated. Clients must re-authenticate.
```

## Code Style

- Run `pnpm lint` before committing (enforced by pre-commit hook)
- Run `pnpm format` to auto-format with Prettier
- Use `@/` path aliases for all imports (never relative paths like `../`)
- Use TypeScript strict mode — avoid `any`

## Adding a New Module

1. Create `src/rest/<module>/` with:
   - `<module>.controller.ts`
   - `<module>.service.ts`
   - `<module>.module.ts`
   - `dto/` folder for request/response DTOs
2. Register the module in `src/rest/modules.ts`
3. Decorate controllers with `@ApiTags`, `@ApiBearerAuth`, `@ApiOperation`, `@ApiResponse`
4. Use `class-validator` on request DTOs and `@ApiProperty` on response DTOs

## Responsible Use of AI

This project permits and encourages the use of AI coding assistants (Claude Code, GitHub Copilot, etc.) with the following guidelines:

### What's expected

- **Understand what you commit.** AI can generate code, but you are responsible for every line that enters the codebase. Read and understand all AI-generated code before committing it.
- **Verify correctness.** AI suggestions can be subtly wrong. Test the code, check edge cases, and validate against the project's existing patterns.
- **Maintain project conventions.** AI output must conform to this project's architecture: DTOs with validation decorators, OpenAPI annotations, `@/` imports, service-layer business logic, etc. Don't commit AI output that ignores established patterns.
- **Don't commit blindly.** If the AI produces code you can't explain to a teammate, don't commit it. Rewrite it until you can.

### What to watch for

- **Security.** AI can introduce injection vulnerabilities, leaked secrets, or insecure defaults. Review all AI-generated code for OWASP top 10 issues. Never commit hardcoded credentials, even as "examples".
- **Hallucinated APIs.** AI may reference functions, methods, or packages that don't exist. Verify that all imports resolve and all called methods exist.
- **Over-engineering.** AI tends to produce abstractions and utilities that aren't needed. Keep it simple — if three lines of straightforward code work, don't accept a 50-line abstraction.
- **License compliance.** Be aware that AI output may resemble copyrighted or licensed code. If the output looks like it was memorized from a specific project, rewrite it.

### Attribution

When a commit is substantially generated by AI, include a `Co-Authored-By` trailer:

```
feat(user): add profile update endpoint

Co-Authored-By: Claude <noreply@anthropic.com>
```

This isn't about policing — it's about traceability. If a bug surfaces later, knowing that a section was AI-generated helps with debugging.

### Academic integrity

This is a school project. AI assistance is a tool, not a substitute for learning. Make sure you can:

1. Explain every architectural decision in the codebase
2. Describe how each module works without looking at the code
3. Debug issues independently when the AI isn't available
4. Justify why you chose a particular approach over alternatives

The goal is to learn software engineering, not to produce code. Use AI to accelerate your learning, not to bypass it.
